Tcl has no enforced project structure. The conventional layout for a Tcl application or library uses a main entry script, a `lib/` or `src/` directory for package source, and a `tests/` directory for test files. Packages are declared with `package provide` and loaded via `package require`. Namespaces organise code logically.

## Example

```
myapp/
├── myapp.tcl            # entry point (source this to run)
├── pkgIndex.tcl         # package index (auto-generated or manual)
├── lib/
│   ├── myapp/
│   │   ├── pkgIndex.tcl # package index for the library
│   │   ├── core.tcl     # core procedures
│   │   └── utils.tcl    # utilities
├── tests/
│   ├── all.tcl          # test runner
│   ├── test_core.tcl    # tcltest-based tests
│   └── test_utils.tcl
└── doc/
    └── myapp.n          # man page
```

```tcl
# lib/myapp/core.tcl
package provide myapp::core 1.0

namespace eval myapp::core {
    proc greet {name} {
        return "Hello, $name!"
    }

    proc version {} { return "1.0.0" }

    namespace export greet version
}

# myapp.tcl — entry point
lappend ::auto_path [file join [file dirname [info script]] lib]
package require myapp::core 1.0

namespace import myapp::core::greet
puts [greet "World"]

# pkgIndex.tcl (generated by pkg_mkIndex or written manually)
# package ifneeded myapp::core 1.0 [list source [file join $dir core.tcl]]
```

```bash
# Generate pkgIndex.tcl automatically
cd lib/myapp
tclsh ../../bin/pkg_mkIndex .
```

## Gotchas

- `pkgIndex.tcl` must exist in the package directory for `package require` to find the package; generate it with `pkg_mkIndex`.
- `info script` returns the path of the currently-executing script — use it as a base path for relative file loading.
- Tcl has no build step; scripts are interpreted directly. "Building" means generating `pkgIndex.tcl` and possibly wrapping with a starpack/starkit.
- Starpacks (`tclsh myapp.tcl` bundled as a single executable) use the `Starkit` infrastructure for deployment.
