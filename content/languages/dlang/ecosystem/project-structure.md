A DUB project has a conventional layout. Source files live under `source/` (default) or `src/`. The entry point is `source/app.d` for executables. Libraries expose their public API through `source/{packagename}/package.d`. Tests live alongside source or in a separate `tests/` directory.

## Example

```
myapp/
├── dub.json            # or dub.sdl — project metadata and deps
├── source/
│   ├── app.d           # main() entry point
│   └── myapp/
│       ├── package.d   # re-exports the public API
│       ├── config.d
│       └── utils.d
├── tests/
│   └── test_utils.d    # unittest blocks or dedicated test runner
└── docs/               # generated by dmd -D or adrdox
```

```d
// source/app.d
module app;

import myapp.utils : greet;
import std.stdio;

void main()
{
    writeln(greet("World"));
}

// source/myapp/utils.d
module myapp.utils;

string greet(string name) { return "Hello, " ~ name ~ "!"; }

unittest
{
    assert(greet("D") == "Hello, D!");
}
```

## Gotchas

- Module names must match the file path: `source/myapp/utils.d` → `module myapp.utils;`.
- DUB automatically picks up all `.d` files under the source directory; no manual file listing is required.
- `package.d` in a directory is the module loaded when you import the directory name (e.g., `import myapp;` loads `myapp/package.d`).
- Keeping `unittest` blocks inside source files (not a separate test directory) is idiomatic; `dub test` compiles and runs them automatically.
