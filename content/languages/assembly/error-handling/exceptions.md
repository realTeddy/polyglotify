Assembly has no exceptions. Error handling is done by checking return values in registers (typically a negative return value or a specific error code in RAX/EAX) or by inspecting a global errno-style variable. Hardware exceptions (divide-by-zero, page fault, invalid opcode) are handled by the OS via signal handlers (POSIX) or Structured Exception Handling (SEH on Windows), not by programmer-defined try/catch blocks.

## Example

```nasm
; Convention: function returns -1 on error, >= 0 on success
; Analogous to C: int result = syscall(...); if (result < 0) { handle_error }

section .text
    global _start

safe_divide:
    ; rdi = a, rsi = b  → returns a/b or -1 if b==0
    test rsi, rsi
    jz   .div_by_zero
    mov  rax, rdi
    cqo                  ; sign-extend rax into rdx:rax
    idiv rsi
    ret
.div_by_zero:
    mov  rax, -1
    ret

_start:
    mov  rdi, 10
    mov  rsi, 0
    call safe_divide
    cmp  rax, -1
    je   handle_error

    ; success path
    jmp  done

handle_error:
    ; "catch" block — write error message, exit with non-zero code
    mov  rax, 60
    mov  rdi, 1          ; exit code 1 = error
    syscall

done:
    mov  rax, 60
    xor  rdi, rdi
    syscall
```

## Gotchas

- There is no stack unwinding — cleanup code (freeing memory, closing files) must be called manually in every error path.
- Hardware exceptions that reach the OS default handler terminate the process; registering a signal handler or SEH frame is the only way to intercept them.
- C++ exception unwinding tables (`.eh_frame`, `LSDA`) are generated by the compiler and live alongside assembly code; hand-written assembly cannot participate in C++ exception handling without manually emitting these tables.
